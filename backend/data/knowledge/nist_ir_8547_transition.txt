NIST IR 8547 — Transition to Post-Quantum Cryptography Standards
Source: NIST Internal Report 8547 (Initial Public Draft), National Institute of Standards and Technology, 2024
Tags: nist, transition, deprecation, timeline, classical_crypto, quantum_safe, sp800-131a

OVERVIEW
NIST IR 8547 provides guidance on the planned deprecation of classical asymmetric cryptographic algorithms and the transition timeline to post-quantum alternatives. It supersedes and extends the guidance in NIST SP 800-131A (Transitioning the Use of Cryptographic Algorithms and Key Lengths).

DEPRECATION SCHEDULE FOR CLASSICAL ALGORITHMS

DISALLOWED after 2030 (immediate action required for long-lived systems):
- RSA with key sizes less than 2048 bits
- Diffie-Hellman (finite field) with less than 2048-bit primes
- Elliptic Curve Diffie-Hellman (ECDH) on curves less than 224 bits

DEPRECATED (use discouraged, migrate by 2030):
- RSA-2048 for key establishment (any purpose)
- RSA-3072 and RSA-4096 for key establishment
- ECDH with any key size (P-256, P-384, P-521, X25519)
- All Diffie-Hellman variants for key exchange
- ECDSA with P-256, P-384, P-521
- RSA-PKCS1v1.5 signatures

WILL REMAIN ALLOWED (quantum-safe or conditionally safe):
- AES-128: Conditionally safe (Grover's gives sqrt speedup; upgrade to AES-256 for highest assurance)
- AES-256: Quantum-safe (recommended)
- SHA-256: Conditionally safe with 128-bit classical security remaining post-quantum
- SHA-384, SHA-512: Quantum-safe
- SHA-3 (SHA3-256, SHA3-384, SHA3-512, SHAKE-128, SHAKE-256): Quantum-safe
- HMAC-SHA256 and stronger: Quantum-safe (symmetric MAC, not affected by Shor's algorithm)

NEW APPROVED ALGORITHMS (from FIPS 203, 204, 205):
- ML-KEM (FIPS 203): Approved for key establishment in all security categories
- ML-DSA (FIPS 204): Approved for digital signatures in all security categories
- SLH-DSA (FIPS 205): Approved for digital signatures in all security categories

TIMELINE FOR FEDERAL/REGULATED ENTITIES

2024:
- FIPS 203, 204, 205 published — organizations should begin planning
- Inventory of classical cryptographic usage required for all federal systems
- Begin piloting PQC in non-production environments

2025:
- No new systems should deploy classical-only key establishment (RSA, ECDH, DH)
- Hybrid PQC (classical + PQC simultaneously) is acceptable and encouraged
- HSM vendors must provide PQC-capable firmware updates

2026:
- All externally-facing systems must support PQC or hybrid key exchange
- New certificates issued by CAs should support hybrid or PQC signing
- Testing and validation of PQC implementations across all systems

2027-2028:
- Migration of internal systems to PQC
- Classical-only TLS cipher suites removed from external services
- PKI root CA migration to PQC signing algorithms

2030:
- RSA and ECC key establishment DISALLOWED for new cryptographic operations
- RSA and ECC signatures deprecated for all new use cases
- Systems unable to migrate must have approved compensating controls

2035 (target):
- Complete deprecation of all classical asymmetric cryptography
- Full PQC ecosystem in production

IMPACT ON SPECIFIC PROTOCOLS

TLS/HTTPS:
- Disallow TLS 1.2 handshakes using RSA/ECDH key exchange by 2030
- TLS 1.3 with hybrid PQC key exchange (X25519MLKEM768) is the interim target
- TLS 1.3 with pure ML-KEM key exchange is the final target

SSH:
- RSA host keys deprecated; migrate to ML-DSA or Ed25519 (Ed25519 is not quantum-safe but has good classical security; ML-DSA-65 recommended for PQC environments)
- Key exchange algorithms using ECDH deprecated; hybrid PQC SSH available via OpenSSH 9.0+

PKI and X.509:
- Root CAs must migrate to ML-DSA-87 or SLH-DSA-SHA2-256s by 2028
- Intermediate CAs: ML-DSA-65 recommended
- End-entity certificates: ML-DSA-65 or hybrid

S/MIME and Email Security:
- S/MIME with RSA encryption deprecated by 2030
- PQC S/MIME profiles being developed by IETF (draft-ietf-lamps-pq-smime)

CODE SIGNING:
- RSA code signing deprecated; migrate to ML-DSA-65 or SLH-DSA-SHA2-128s by 2027

KEY MANAGEMENT AND HSMS:
- HSM vendors must provide PQC-capable key generation and storage by 2026
- Key wrapping algorithms must use AES-256-GCM minimum
- Key derivation functions: HKDF-SHA-256 acceptable; HKDF-SHA3-256 preferred

QUANTUM RISK CATEGORIES UNDER NIST IR 8547
Level 0 (No Risk): Symmetric cryptography with ≥128-bit keys, hash functions with ≥256-bit output
Level 1 (Low Risk): Classical asymmetric with key exchange, data sensitivity lifetime < 5 years, no HNDL exposure
Level 2 (Medium Risk): Classical asymmetric, data lifetime 5-10 years, or externally exposed
Level 3 (High Risk): Classical asymmetric, data lifetime >10 years, or HNDL-exposed financial data
Level 4 (Critical): Classical key exchange protecting data that is actively harvested, CRQC arrival uncertainty, mission-critical

IMPLEMENTATION PRIORITY ORDER (NIST RECOMMENDATION)
1. Discontinue use of RSA/ECC for key establishment in externally exposed services
2. Enable hybrid TLS (X25519MLKEM768) as a transition measure
3. Update certificate signing chains to PQC algorithms
4. Migrate key management infrastructure (HSMs, KMS)
5. Update application-level cryptographic libraries
6. Deprecate classical-only cipher suites from all TLS configurations
