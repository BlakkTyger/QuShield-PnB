NIST FIPS 204 — Module-Lattice-Based Digital Signature Algorithm Standard (ML-DSA)
Source: National Institute of Standards and Technology, August 2024
Tags: pqc, digital_signature, dilithium, lattice, fips204, nist, quantum_safe

OVERVIEW
NIST FIPS 204 standardizes ML-DSA (formerly CRYSTALS-Dilithium), a lattice-based digital signature scheme based on the hardness of the Module Learning With Errors (MLWE) and Module Short Integer Solution (MSIS) problems. It is designed to replace RSA signatures, ECDSA, and EdDSA in post-quantum deployments.

PURPOSE AND USE CASES
ML-DSA provides authentication, integrity, and non-repudiation. Key applications:
- TLS certificate signatures (replacing RSA-SHA256, ECDSA-SHA256)
- Code signing and software integrity verification
- JWT and API authentication tokens
- Document signing and audit trails
- Inter-bank transaction signing (RTGS, NEFT, IMPS authorization)

PARAMETER SETS
- ML-DSA-44: NIST Security Level 2. Public key: 1312 bytes. Signature: 2420 bytes.
- ML-DSA-65: NIST Security Level 3. Public key: 1952 bytes. Signature: 3293 bytes. RECOMMENDED for general use.
- ML-DSA-87: NIST Security Level 5. Public key: 2592 bytes. Signature: 4595 bytes. For highest-assurance environments (central bank, cross-border settlements).

SECURITY PROPERTIES
- Existentially Unforgeable under Chosen Message Attack (EUF-CMA secure)
- Quantum-safe against Shor's algorithm and Grover's algorithm
- Deterministic signing: eliminates randomness-related vulnerabilities (unlike ECDSA which requires high-quality random nonces)
- Side-channel resistant design

COMPARISON WITH CLASSICAL SIGNATURES
- RSA-2048 signature: 256 bytes. ML-DSA-65: 3293 bytes (12.9x larger).
- RSA-2048 verification: very fast. ML-DSA verification: ~3x slower than RSA but acceptable for most use cases.
- ECDSA-P256 signature: 64 bytes. ML-DSA-65: 3293 bytes (51x larger). This impacts TLS certificate sizes.
- Performance: ML-DSA signing is faster than RSA signing; verification is slower than ECDSA.

BANKING-SPECIFIC IMPLEMENTATION NOTES
1. Certificate sizes: PQC certificates with ML-DSA are significantly larger — TLS handshake size increases from ~3KB to ~8-12KB. Ensure load balancers and WAFs support larger handshakes.
2. JWT tokens signed with ML-DSA will be much larger — consider ML-DSA-44 for short-lived access tokens.
3. HSM support: Verify HSM vendor supports ML-DSA before planning PKI migration. Thales, Utimaco, and nCipher have PQC HSM roadmaps.
4. Certificate lifecycle: Plan for shorter certificate validity periods during hybrid transition (RSA+ML-DSA dual-signed certificates are a transition option).
5. Code signing for core banking software: High priority — use ML-DSA-87 for software update signatures.

HYBRID SIGNATURES
During the transition period, dual-signing (classical + PQC) is recommended:
- X.509 certificates can carry both ECDSA and ML-DSA signatures (composite signatures per draft-ietf-lamps-pq-composite-sigs)
- This ensures compatibility with both classical and PQC-capable verifiers

RBI AND REGULATORY COMPLIANCE
- RBI IT Framework 2023: Requires quantum-resistant signing for inter-bank settlement messages by 2027
- NPCI Guidelines: UPI and payment system digital signatures must migrate to PQC algorithms per NPCI circular 2024
- SEBI CSCRF: Capital market systems must implement PQC signatures for trade confirmations and audit logs
- PCI DSS 4.0: Requires cryptographic agility supporting algorithm migration

MIGRATION PRIORITY
CRITICAL: PKI Certificate Authority root and intermediate CA signatures
HIGH: TLS server certificates for payment processing endpoints
HIGH: Transaction authorization and settlement message signing
MEDIUM: API authentication tokens (JWTs, HMAC-based auth)
LOW: Internal logging and monitoring signatures
