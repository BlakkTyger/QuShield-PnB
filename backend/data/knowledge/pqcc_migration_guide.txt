OVERVIEW
The Post-Quantum Cryptography Coalition (PQCC) is an industry-led initiative (founding members: Microsoft, IBM, MITRE, NIST, Amazon, Google, Cisco, Intel) providing practical migration guidance for enterprises transitioning from classical to post-quantum cryptography. The PQCC methodology defines a 4-phase migration framework applicable to any organization.

THE 4-PHASE PQCC MIGRATION FRAMEWORK

PHASE 1: INVENTORY AND DISCOVERY
Duration: 3-6 months
Objective: Create a complete, accurate picture of all cryptographic assets.

Key Activities:
1.1 Network scanning for TLS/SSL cipher suites across all exposed services
1.2 Certificate inventory — document all public and internal certificates (type, key length, algorithm, expiry, issuer CA)
1.3 Code analysis — static analysis of application source code for hard-coded crypto calls
1.4 Dependency scanning — identify cryptographic libraries (OpenSSL, BouncyCastle, NSS, SunJCE) in all applications
1.5 HSM and key management inventory — document all keys in HSMs, key stores, and cloud KMS
1.6 Third-party vendor assessment — survey all technology vendors for their cryptographic dependencies
1.7 Data classification — identify which data stores contain sensitive data with long shelf lives (>5 years)
1.8 CBOM generation — produce machine-readable Cryptographic Bill of Materials in CycloneDX 1.6 format

Deliverables:
- Complete CBOM in CycloneDX format
- List of all unique cryptographic algorithms in use (with counts and locations)
- Dependency map of cryptographic libraries
- Vendor quantum readiness survey responses
- Data sensitivity and retention classification

1.9 Common Findings During Inventory:
- RSA-2048 or smaller used for key exchange (quantum vulnerable)
- ECDSA P-256 or P-384 for signatures (quantum vulnerable)
- TLS 1.2 still active on internal services
- Expired or soon-to-expire certificates
- Hard-coded keys in application configs
- HSMs not supporting PQC algorithms

PHASE 2: PRIORITIZE AND PLAN
Duration: 2-4 months
Objective: Risk-rank all cryptographic assets and create a detailed migration roadmap.

Key Activities:
2.1 Apply Mosca's Inequality to each asset class:
    Risk = (Migration Time) + (Data Sensitivity Lifetime) > (CRQC Arrival Estimate)
    If Risk > 0: URGENT (migrate now or within 1 year)
    If Risk ≈ 0: HIGH (migrate within 2 years)
    If Risk < 0: MEDIUM/LOW (migrate within 3-5 years)

2.2 HNDL (Harvest Now, Decrypt Later) assessment:
    Identify which data transmitted today could be decrypted by a CRQC in the future.
    Priority 1: Long-term sensitive data (medical records, legal documents, national security data)
    Priority 2: Financial transaction archives (>5 year retention)
    Priority 3: Authentication credentials stored encrypted

2.3 System complexity assessment:
    - API surface area (number of public TLS endpoints)
    - Number of dependent systems (blast radius if crypto fails during migration)
    - Legacy system constraints (cannot easily update protocol stacks)
    - HSM vendor support timeline
    - CA (Certificate Authority) migration complexity

2.4 Define migration cohorts:
    Cohort A: Externally facing, high HNDL risk → migrate in 0-12 months
    Cohort B: Internal high-sensitivity, medium HNDL risk → migrate in 12-24 months
    Cohort C: Internal low-sensitivity, low complexity → migrate in 24-36 months
    Cohort D: Legacy/constrained systems → special handling, extended timeline

Deliverables:
- Prioritized migration roadmap with cohorts and timelines
- Mosca analysis report per asset class
- Vendor negotiation plan for HSM/CA/library upgrades
- Budget and resource estimation
- Risk acceptance register for systems that cannot be migrated on schedule

PHASE 3: MIGRATE
Duration: 12-36 months (depends on organization size)
Objective: Implement PQC algorithms in production systems.

Key Activities:
3.1 Algorithm selection per use case:
    - Key exchange: ML-KEM-768 (or hybrid X25519MLKEM768 during transition)
    - Signatures: ML-DSA-65 for performance-sensitive; SLH-DSA-SHA2-128s for root CA
    - Symmetric: AES-256 (already quantum-safe with Grover resistance; 256-bit keys required)
    - Hash: SHA-3 (SHAKE-256, SHA3-256) preferred; SHA-256 acceptable with doubled key sizes

3.2 TLS migration steps:
    a) Enable hybrid X25519MLKEM768 key exchange (backward compatible with classical clients)
    b) Update server certificates to hybrid or PQC-only certificates (once CA support available)
    c) Update cipher suite priority lists to prefer PQC algorithms
    d) Test and validate with both PQC-capable and legacy clients

3.3 PKI migration:
    a) Migrate offline root CA to PQC signing algorithm (ML-DSA-87 or SLH-DSA-SHA2-256s recommended for root)
    b) Issue intermediate CAs with hybrid dual-algorithm certificates
    c) Roll out end-entity certificates with PQC algorithms
    d) Monitor CT logs for unexpected certificate issuance

3.4 Application-level migration:
    a) Update cryptographic libraries (OpenSSL → v3.4+ with OQS provider, BouncyCastle → v2.0+)
    b) Refactor hard-coded algorithm references to configuration-driven implementations
    c) Update JWT signing to use ML-DSA (or keep HS256 which is quantum-safe at ≥256 bits)
    d) HSM firmware updates — coordinate vendor-specific timelines

3.5 Hybrid transition strategy (RECOMMENDED for banking):
    - Run classical + PQC algorithms in parallel for 12-24 months
    - Classical algorithms ensure backward compatibility with legacy clients
    - PQC algorithms protect against quantum threats for all PQC-capable clients
    - Monitor traffic mix: when >95% of connections are PQC-capable, classical can be deprecated

Deliverables:
- Updated system configurations with PQC algorithms
- New certificate inventory (PQC-signed)
- Updated dependency versions
- Performance benchmarking results (before/after)
- Rollback procedures for each system

PHASE 4: VERIFY AND MONITOR
Duration: Ongoing (3-6 months initial verification + continuous monitoring)
Objective: Confirm migration is complete and effective; maintain posture over time.

Key Activities:
4.1 Cryptographic posture verification:
    - Re-scan all external surfaces for remaining classical-only endpoints
    - Validate TLS handshakes are using PQC or hybrid key exchange
    - Verify certificate chain uses PQC signing algorithms
    - Check no regression to classical algorithms (firewall rules, load balancer configs)

4.2 Compliance validation:
    - Run compliance checks against NIST SP 800-131A Rev 3 deprecation schedules
    - Validate against RBI IT Framework 2023 requirements
    - Prepare audit evidence package (CBOM, scan results, configuration snapshots)

4.3 Continuous monitoring:
    - Implement cryptographic algorithm alerting (alert if classical-only TLS connections detected)
    - Integrate CBOM into CI/CD pipeline (scan every build for new cryptographic dependencies)
    - Subscribe to NIST/CERT alerts for PQC algorithm vulnerabilities
    - Annual re-assessment of migration completeness

4.4 Crypto-agility maintenance:
    - Document procedure to swap algorithms within 72 hours if vulnerability discovered
    - Quarterly crypto-agility drills (tabletop exercises)
    - Algorithm deprecation calendar maintained and reviewed annually

COMMON MIGRATION MISTAKES TO AVOID
1. Migrating only public-facing services — internal mTLS and service mesh are often overlooked
2. Forgetting key management infrastructure (HSMs, key vaults) — these need PQC support too
3. Assuming library upgrade = migration complete — application code may still hard-code classical algorithms
4. Not testing with real traffic — lab testing misses edge cases in protocol negotiation
5. Ignoring certificate size increases — PQC certificates are 3-10x larger; update firewall and WAF limits
6. Not coordinating with third-party vendors — vendor lock-in can block migration timelines
7. Skipping hybrid mode — going directly from classical to PQC-only breaks legacy clients
