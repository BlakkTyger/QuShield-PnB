NIST FIPS 203 — Module-Lattice-Based Key-Encapsulation Mechanism Standard (ML-KEM)
Source: National Institute of Standards and Technology, August 2024
Tags: pqc, key_encapsulation, kyber, lattice, fips203, nist, quantum_safe

OVERVIEW
NIST FIPS 203 standardizes ML-KEM (formerly known as CRYSTALS-Kyber), a key-encapsulation mechanism based on the hardness of the Module Learning With Errors (MLWE) problem. It is one of three initial post-quantum cryptographic standards published by NIST in August 2024 alongside FIPS 204 (ML-DSA) and FIPS 205 (SLH-DSA).

PURPOSE AND USE CASES
ML-KEM is designed for key establishment — specifically, to allow two parties to establish a shared secret over an insecure channel. It replaces classical key exchange mechanisms such as RSA key transport, Diffie-Hellman (DH), and Elliptic Curve Diffie-Hellman (ECDH). ML-KEM is NOT a digital signature scheme. Use ML-DSA or SLH-DSA for signatures.

PARAMETER SETS
FIPS 203 defines three parameter sets offering different security/performance tradeoffs:
- ML-KEM-512: NIST Security Level 1 (comparable to AES-128). Public key: 800 bytes. Ciphertext: 768 bytes. Shared secret: 32 bytes.
- ML-KEM-768: NIST Security Level 3 (comparable to AES-192). Public key: 1184 bytes. Ciphertext: 1088 bytes. Shared secret: 32 bytes. RECOMMENDED for most applications.
- ML-KEM-1024: NIST Security Level 5 (comparable to AES-256). Public key: 1568 bytes. Ciphertext: 1568 bytes. Shared secret: 32 bytes. Use for highest-assurance environments.

SECURITY PROPERTIES
- Indistinguishable under Chosen Ciphertext Attack (IND-CCA2 secure)
- Quantum-safe: No known polynomial-time quantum algorithm can break MLWE
- Classical security: Relies on hardness of lattice problems (LWE, MLWE)
- Forward secrecy: When used in ephemeral key exchange (KEM+symmetric encryption)

TLS INTEGRATION
ML-KEM-768 is the primary PQC key exchange algorithm for TLS 1.3. The hybrid X25519MLKEM768 group (combining classical X25519 with ML-KEM-768) is specified in the IETF draft-ietf-tls-hybrid-design. This hybrid approach provides protection against both classical and quantum adversaries simultaneously and is the RECOMMENDED transition mechanism. Browsers including Chrome and Firefox have enabled X25519MLKEM768 by default as of 2024-2025.

IMPLEMENTATION GUIDANCE FOR BANKING INFRASTRUCTURE
1. Replace RSA key exchange (RSA-2048, RSA-4096) and ECDH (P-256, P-384, X25519) with ML-KEM-768 or hybrid X25519MLKEM768.
2. Prioritize HTTPS endpoints, API gateways, and inter-service mTLS connections.
3. For RBI compliance: ML-KEM-768 at minimum is required for data classified as "sensitive" or "critical" under RBI IT Framework 2023.
4. Certificate pinning with ML-KEM keys requires CA support — verify CA PQC readiness before deployment.
5. Key sizes are larger than classical: plan for increased TLS handshake overhead (~1-2ms additional latency for ML-KEM-768 vs X25519).

MIGRATION PRIORITY
HIGH: All TLS 1.3 key exchange for external-facing services (internet-exposed APIs, payment gateways).
HIGH: Inter-service communication carrying financial data.
MEDIUM: Internal management interfaces.
LOW: Non-sensitive internal monitoring traffic.

COMPLIANCE MAPPING
- NIST SP 800-208: ML-KEM-768 satisfies quantum-resistant key establishment requirements
- NIST SP 800-131A Rev 3: Classical key exchange (RSA, DH, ECDH) deprecated for sensitive data after 2030
- RBI IT Framework 2023 Annex A: Quantum-safe cryptography mandated for all inter-bank and customer-facing channels
- PCI DSS 4.0 Requirement 4.2.1: Strong cryptography required — ML-KEM satisfies this
- ISO/IEC 18033-3: Key encapsulation mechanisms standard (ML-KEM is compatible)
