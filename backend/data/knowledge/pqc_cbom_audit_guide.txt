Cryptographic Bill of Materials (CBOM) — Audit Guide and CycloneDX Standard
Source: CycloneDX CBOM Specification v1.6 (OWASP), NIST SP 800-218A, CISA CBOM Guidance
Tags: cbom, cyclonedx, audit, inventory, algorithm, cryptographic_bom, pqc

WHAT IS A CBOM?
A Cryptographic Bill of Materials (CBOM) is a machine-readable inventory of all cryptographic assets in a software system, analogous to a Software Bill of Materials (SBOM) but focused specifically on cryptographic algorithms, protocols, keys, and certificates. CycloneDX 1.6+ includes native CBOM support.

CBOM COMPONENTS (per CycloneDX 1.6 specification)

1. Algorithms
Each distinct cryptographic algorithm instance documented with:
- algorithmRef: Unique identifier (e.g., "RSA-2048-key-exchange")
- primitive: Type of cryptographic primitive (hash, symmetric, asymmetric-encryption, digital-signature, key-agree, key-derive)
- parameterSetIdentifier: Key length, curve name, or hash output size
- curve: For ECC algorithms (P-256, P-384, P-521, X25519, X448, BrainpoolP256r1)
- executionEnvironment: Where algorithm runs (software, hardware-HSM, firmware, enclave)
- implementationPlatform: Library or hardware implementing it (openssl, bouncycastle, nss, pkcs11)
- nistQuantumSecurityLevel: NIST security level (0-6; -1 if unknown/vulnerable)
  - Level 0: Quantum-vulnerable (RSA, ECDSA, ECDH, DH with any key size)
  - Level 1: Equivalent to AES-128 post-quantum security
  - Level 3: Equivalent to AES-192 post-quantum security
  - Level 5: Equivalent to AES-256 post-quantum security

2. Certificates
X.509 certificates documented with:
- subjectName (Common Name, SANs)
- issuerName (CA that issued the certificate)
- algorithm: Signature algorithm (sha256WithRSAEncryption, ecdsa-with-SHA256, id-ML-DSA-65)
- serialNumber
- notBefore, notAfter (validity period)
- subjectPublicKeyInfo: Key type and length
- signatureValue
- quantumVulnerable: Boolean flag

3. Protocols
TLS, SSH, and other protocol configurations documented with:
- type: tls, ssh, ipsec, ike, pkix
- version: 1.0, 1.1, 1.2, 1.3, 2.0
- cipherSuites: List of negotiated cipher suites
- keyExchangeAlgorithm: Component ref to key exchange algorithm
- signatureAlgorithm: Component ref to signature algorithm
- quantumSafe: Boolean flag (true only if PQC or hybrid key exchange)

4. Keys
Cryptographic keys documented with:
- keyType: RSA, EC, X25519, ML-KEM, ML-DSA, AES, HMAC
- size: Key size in bits
- format: PKCS8, PKCS12, JWK, PEM, DER
- secured: Boolean (stored in HSM or secure vault)
- quantumSafe: Boolean

ALGORITHM QUANTUM VULNERABILITY CLASSIFICATION

QUANTUM VULNERABLE (NIST Quantum Level 0 — must migrate):
- RSA (any key size) — broken by Shor's algorithm
- ECDSA (P-256, P-384, P-521, K-163) — broken by Shor's algorithm
- ECDH, X25519, X448 — broken by Shor's algorithm
- DSA (finite-field) — broken by Shor's algorithm
- Diffie-Hellman (any finite-field group) — broken by Shor's algorithm
- ElGamal — broken by Shor's algorithm

CONDITIONALLY QUANTUM SAFE (symmetric, not broken by Shor's):
- AES-128: Grover's algorithm gives sqrt speedup → effectively 64-bit classical equivalent. Use AES-256 for long-term security.
- AES-256: Quantum-safe. Recommended.
- SHA-256: Grover's gives 128-bit classical equivalent. Acceptable for most uses.
- SHA-384, SHA-512: Quantum-safe.
- HMAC-SHA256: Quantum-safe (symmetric MAC).
- HMAC-SHA512: Quantum-safe.
- SHA-3 family: Quantum-safe.

POST-QUANTUM ALGORITHMS (NIST Levels 1-5 — quantum-safe):
- ML-KEM-512 (Kyber512): Level 1
- ML-KEM-768 (Kyber768): Level 3 — RECOMMENDED
- ML-KEM-1024 (Kyber1024): Level 5
- ML-DSA-44 (Dilithium2): Level 2
- ML-DSA-65 (Dilithium3): Level 3 — RECOMMENDED
- ML-DSA-87 (Dilithium5): Level 5
- SLH-DSA-SHA2-128s: Level 1
- SLH-DSA-SHA2-192s: Level 3
- SLH-DSA-SHA2-256s: Level 5
- X25519MLKEM768 (hybrid): Level 3 — RECOMMENDED for TLS transition

CBOM AUDIT FINDINGS CLASSIFICATION
Critical: Quantum-vulnerable algorithm used for key exchange on externally accessible endpoint
High: Quantum-vulnerable algorithm used for signing long-lived certificates (>1 year validity)
Medium: TLS 1.2 with classical cipher suite used for internal service communication
Low: Quantum-vulnerable algorithm used for non-sensitive, short-lived data
Informational: Algorithm is quantum-safe but not the highest security level available

CBOM AUDIT PROCESS FOR BANKING INFRASTRUCTURE
1. Network scan: Identify all TLS endpoints and negotiate cipher suites
2. Certificate extraction: Parse all X.509 certificates from discovered services
3. Certificate chain analysis: Trace CA chain, identify root/intermediate signing algorithms
4. Protocol inventory: Document TLS versions, key exchange groups, cipher suites per endpoint
5. Algorithm extraction: Map each cipher suite to its constituent algorithms (key exchange + authentication + encryption + MAC)
6. Quantum classification: Apply NIST quantum level to each algorithm instance
7. CBOM generation: Produce CycloneDX 1.6 JSON with all components tagged
8. Vulnerability scoring: Apply QuShield quantum risk scores based on algorithm classification and context

CBOM REPORTING METRICS
- Total components: Total distinct cryptographic component instances
- Quantum-vulnerable components: Count with NIST quantum level = 0
- Quantum-ready components: Count with NIST quantum level >= 1
- Quantum-ready percentage: (quantum_ready / total) * 100
- Top algorithms: Most frequently used algorithms (by count)
- Algorithm diversity index: Measure of cryptographic agility (more diverse = more agile)
